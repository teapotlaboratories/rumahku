# M1 — Integrate Brush's training core into rumahku

Goal: after a scan, reconstruct a Gaussian splat **on the phone** by calling
Brush's training core from the rumahku app over JNI, producing a `.ply`. This
turns the M0 feasibility result (Brush trains on the Pixel 6 Mali GPU) into an
in-app feature.

Prereq context: see `docs/PHASE2.md` (overall Phase 2 plan) and the M0 result
in `docs/worklog/2026-07-07.md` (Brush trains a splat on the Pixel 6: ~5.5
it/s, 311 MB, 32 °C; needs a **writable kernel-cache dir**).

---

## 1. Architecture fit

Brush's core is **stream-based**. `brush-process` exposes a `ProcessStream`
(`Stream<Item = Result<ProcessMessage, Error>>`), and `run_headless()` drives it
to completion. The stream emits:

- `ProcessMessage::StartLoading` / `DoneLoading`
- `ProcessMessage::SplatsUpdated { .. }` — the current splat, at any step
- `ProcessMessage::TrainMessage(TrainMessage::{TrainConfig, Dataset, TrainStep, RefineStep, DoneTraining, EvalResult})`

So the FFI simply wraps a `run_headless`-like loop with a **C/JNI callback**
instead of the terminal UI. Two consequences:

- **Input handoff is nearly free.** The app already writes nerfstudio
  `transforms.json` + JPEGs; Brush reads that directory directly as a
  `DataSource::Path`. No conversion.
- **M2 comes for free later.** The same `SplatsUpdated` stream that lets M1
  export a final `.ply` also exposes per-step splats for a live preview (M2) —
  no rework of the core.

---

## 2. The FFI surface — `brush-ffi` (new `cdylib` crate in a Brush fork)

Expose native methods to Kotlin via the `jni` crate (Rust functions named
`Java_com_teapotlab_rumahku_...`), so no separate C shim is needed:

- `nativeInit(cacheDir: String)` — set cubecl's cache dir to app-writable
  storage (the M0 finding) and init the wgpu/Vulkan device once.
- `nativeTrain(datasetDir, outPlyPath, totalIters, maxRes, maxFrames): String`
  — run the training stream to completion on a worker thread; export the `.ply`;
  return the output path (or an error string).
- **progress callback** into Kotlin: `onStep(iter, splatCount, elapsedMs)`, fed
  from `TrainStep` / `RefineStep`.
- `nativeCancel()` — cooperative abort.

Internals mirror `apps/brush-cli`'s `run_headless`: build a tokio runtime on a
dedicated thread, init the device, drive the `ProcessStream`, translate messages
to callbacks, and on `DoneTraining` write the splat to `.ply`.

---

## 3. Build & app wiring

- Build: `cargo ndk -t arm64-v8a -o app/src/main/jniLibs build --release -p brush-ffi`
  → `libbrush_ffi.so` placed under `app/src/main/jniLibs/arm64-v8a/`.
- Kotlin `BrushTrainer` object: `System.loadLibrary("brush_ffi")` + `external
  fun` declarations; call `nativeTrain` from a coroutine (off the main thread)
  inside a **foreground service** (long GPU compute; hold a wakelock; show
  progress + allow cancel).
- App flow: after "Start scanning" produces a capture directory
  (`transforms.json` + images), add a **"Reconstruct"** action →
  `BrushTrainer.train(captureDir)` → `.ply` → show/export the result.
- Pass `context.cacheDir` (app-private, writable) as the cubecl cache dir.

---

## 4. Risks (ranked)

1. **`.so` size — RESOLVED by the size spike (2026-07-07).** Landed at **44 MB
   stripped** (from 94 MB), with **no training-speed regression** (5.4 vs 5.5
   it/s on the Pixel 6). Key learning: the bulk is the core ML/GPU stack
   (`.rodata` shader/kernel data ~30 MB + `.text`), **not** rerun (cutting rerun
   saved only ~5 MB). The lever was the **release profile**, not dependency
   cuts: `opt-level="s"` + `lto="fat"` + `codegen-units=1` + `panic="abort"`
   (the latter is also correct for an FFI lib). `panic="abort"` removed ~7 MB of
   unwind tables; the profile cut `.text` 51→11 MB. `opt-level="z"` is
   unnecessary (`"s"` already small, and safer for the small amount of hot CPU
   code). See §7.
2. **`jni` crate version.** The `brush-app` build failed in `rrfd` on a `jni`
   0.22 API split (`Env` / `EnvUnowned`). `brush-ffi` must pin a `jni` version
   that compiles cleanly against Brush's tree — settle this up front.
3. **Cache dir plumbing.** Point cubecl's cache at `context.cacheDir` (env var
   or API — confirm during impl). Without it, every cold start re-autotunes
   (~40 s/kernel per M0).
4. **In-process GPU access.** Running inside the app (not the adb shell) should
   *improve* GPU access vs. M0, but validate device init succeeds in an app
   process with the app's permissions.
5. **Long-run memory/thermals.** M0 was deliberately tiny (25 views, 640px, 300
   iters). Real scans (more views, higher res, more iters) need memory/thermal
   validation; caps + downscale live in M5 but keep them in mind here.

---

## 5. Fork strategy

Prefer a **thin `brush-ffi` on top of a pinned upstream Brush** (pin the working
commit `3b80985`) rather than a hard fork, to keep upstream pulls easy — matches
`docs/PHASE2.md`. Vendor Brush as a submodule or pinned path/git dependency.

---

## 6. Task sequence

1. Fork / vendor Brush pinned at `3b80985`; add a `crates/brush-ffi` skeleton
   (`crate-type = ["cdylib"]`).
2. **Size spike** — build a stripped, rerun-less cdylib and measure the `.so`.
   If it can't be trimmed to a shippable size, that reshapes M1.
3. Implement `nativeInit` / `nativeTrain` + progress callback; build into
   `app/src/main/jniLibs/`.
4. Kotlin `BrushTrainer` + foreground service; wire the "Reconstruct" action.
5. End-to-end: scan a room in-app → reconstruct on-device → view/share `.ply`.

Exit criterion: from a scan captured by the app, the app produces a `.ply`
on-device via Brush, with visible progress and no crash.

---

## 7. Outcome

**Size spike — done (2026-07-07).** On a Brush fork branch `rumahku-ffi`
(pinned at `3b80985`, in `~/Developments/rumahku-splat/brush`):
- Excluded `rerun` for the Android target by treating Android like WASM (reused
  Brush's existing no-op `VisualizeTools`): gated `crates/brush-rerun/src/{
  visualize_tools.rs,lib.rs}` and the `[target.'cfg(...)'.dependencies]` rerun
  block on `not(target_os = "android")`. Saved only ~5 MB (rerun is build-time
  heavy, size-light) but also trims build time.
- Added `[profile.release-small]` (inherits release; `opt-level="s"`,
  `lto="fat"`, `codegen-units=1`, `panic="abort"`, `strip=true`).
- Build: `cargo ndk -t arm64-v8a build --profile release-small -p brush-cli`.
- Result: **44 MB stripped** (from 94 MB no-rerun / 99 MB baseline), sections
  `.rodata` 30 MB + `.text` 11 MB. On-device train unchanged: 200 iters in 37 s
  (**5.4 it/s**), valid `.ply`. → **Size risk retired; shippable `.so`.**

Still to do: FFI crate, JNI, Kotlin wiring, end-to-end (§6 tasks 3–5).
