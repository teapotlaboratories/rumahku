# Worklog — 2026-07-12

Two threads: (1) PSNR reporting honesty, (2) on-device resolution auto-adjust.

## Thread 1 — raw-PSNR honesty (backend + app)

**Problem.** The per-scan quality score preferred `cc_psnr` (colour-corrected).
cc applies a per-view affine colour match that hides the bilateral grid
(train-only, stripped at export) + auto-exposure drift, so it **overstates what
the exported `.ply` actually renders**. See `docs/design/quality-capture-mode.md`.

**Finding while investigating.** The three splatfacto jobs currently retained on
the backend have `metrics.json` with only `psnr` — **no `cc_psnr` key at all**
(`res.get("cc_psnr", res.get("psnr"))` was already silently falling through to
raw). One good capture even hit raw 24.68. So the flip mostly makes raw-first
*intentional and robust* rather than fixing a live overstatement — but it still
matters for any bilateral-grid run that *does* emit cc.

**Change.**
- `serve.py` `_read_ns_metrics`: headline = raw `psnr`/`ssim`/`lpips`; `cc_*`
  kept only as a diagnostic when present. Status endpoint returns `cc_psnr` only
  when present. `_quality_verdict` bands documented as the raw regime (Brush
  already reported raw, so verdicts are now consistent across trainers).
- App: `CloudBuild.kt` parses/persists optional `cc_psnr`; the metrics dialog
  shows a "PSNR · colour-matched" row **only when present**, labeled a train-only
  reference, and the footnote clarifies the headline is raw held-out PSNR.

**Validated** by replaying the exact logic over real + synthetic metrics: real
(no cc) → headline 19.18, no cc row; a bilateral-grid job (raw 18.1 / cc 23.5) →
headline **18.1** (not the flattering 23.5), cc 23.5 surfaced as diagnostic.

## Thread 2 — on-device resolution auto-adjust (app)

**Goal.** Raise on-device Brush reconstruction above the hardcoded 720p, and
auto-adjust the ceiling per device (different GPUs/RAM → different ceilings).

**Key constraint.** The Brush FFI is built `panic=abort`, so a GPU OOM at higher
resolution is usually a **hard process crash** (SIGABRT / kernel SIGKILL), not a
catchable `"ERROR:"` string. So auto-adjust must be **crash-survivable**, not a
simple in-process retry.

**Design — "table + crash-learning"** (`DeviceCapability.kt`, app-only; the FFI
already takes `maxRes`):
- RAM-tier table → safe default (Pixel 6 8 GB → 1024, ~12 GB → 1280, etc.).
- Before each train, the attempted resolution is **committed to disk**
  (`attempt_res`). A clean return clears it. If the process dies, the marker
  survives → the next run detects a crash and lowers the ceiling.
- Learning is asymmetric by confidence: a crash in **new territory** (above the
  proven-good res) drops one rung immediately; a crash **at/below** a proven-good
  res needs 2 consecutive strikes (so one spurious external kill doesn't cost a
  rung). A confirmed OOM **decays** the proven-good res below the failing rung, so
  the ceiling actually converges (invariant: `known_good ≤ stepBelow(crash_ceiling)`).
- Ladder `[512,640,720,900,1024,1280]`; learned ceiling only moves **down** from
  the table. A catchable OOM also walks the ladder down in-run.

**On-device OOM sweep (Pixel 6, oriole / Mali-G78, seeded 1080p scene, 2000 iters,
max-frames 40)** — via `brush-cli-small` (= the app's `release-small` profile):

| max_res | result | peak RSS | note |
|---|---|---|---|
| 512  | ok (smoke) | 234 MB | — |
| 1024 | **ok** | 459 MB | 8 GB-tier default |
| 1280 | **ok** | 612 MB | 12 GB-tier default, proven even on this 8 GB device |

Thermals 32 → 38 °C over ~45 min continuous — no throttling. Standalone
`brush-cli` has more headroom than the app process, so these are upper bounds; the
ladder caps at 1280 (no tier selects above it), so 1536/1920 weren't needed.

**In-app validation (Pixel 6, WIP build).** On-device picker now reads
"Brush, 1024p" (was 720p). Starting a Fast build logged
`resolveForRun: ram=1024 known=0 crashCeil=0 → 1024p`; `recon_caps.xml` held
`attempt_res=1024` mid-train (crash marker durably committed before the crash
window); trained at ~567 MB RSS / 36.5 °C with no OOM/crash.

**Adversarial review** (multi-agent) caught 10 real issues in the first cut — most
importantly that `known_good` never decayed, which (a) made a hard crash at exactly
the proven ceiling an unrecoverable loop and (b) let a `crash_ceiling ≤ known_good`
oscillate. Fixed by the decay + strike-counter design above; also: `startForeground`
now precedes every early return (pre-existing FGS-timeout risk), the retry comment
matches the multi-rung behaviour, the progress notification shows the live res, and
assorted cleanups. Thread-1 (metrics) review came back clean.

A **re-review** confirmed all 10 fixed and caught 2 more: `finishAttempt(OTHER)`
(cancel / non-OOM error) now also clears a matching strike — any clean, non-OOM
return proves the run didn't hard-crash, so it breaks the streak (was inconsistent
with SUCCESS); and `plannedRes` takes a `buildRunning` flag so an *in-flight* attempt
marker isn't misread as a crash while a build is running (was a cosmetic wrong label).

**Crash-learning demonstrated live** on the Pixel 6 by injecting the marker state and
reading it back through the picker label (`plannedRes` runs the real reconcile): with
`known_good=1024`, one stale crash at 1024 → label stays **1024p** (one spurious kill
tolerated); a second consecutive stale crash → label drops to **900p** (reproducible
OOM → ceiling steps down). This is exactly the HIGH-severity scenario the first review
flagged, now recovering.
