## Made phone unlock non-flaky
- Root causes of the flaky unlock: (1) phone is in **gesture-nav mode**, so the
  bottom-edge swipe was eaten as a system gesture; (2) digits sent during the
  bouncer raise-animation were dropped; (3) `deviceLocked`/screencap read stale.
- Fix: **`tools/adb-unlock.sh`** — runs an on-device timed sequence (sleeps
  between wake / mid-screen swipe / digits), resets each attempt, retries, and
  polls until unlock settles. Carries NO PIN (reads `$PIXEL_PIN` or
  `~/.config/rumahku/pixel-pin`, mode 600, gitignored). `tools/adb-unlock.sh
  <package>` also launches an app. Tested **4/4 first-attempt** unlock+launch.
  A subtle bug fixed along the way: detection polled `deviceLocked` (which
  settles slower than the real unlock) and each retry re-locked a
  just-unlocked device — now it checks "already unlocked" before re-locking.
  Documented in `docs/BENCH.md`.

## Quality assessment of the M1 splat (before investing in M3)
- Used Brush's own eval on-device (load M1 splat as `ply_file_path` seed, eval
  vs held-out views): **PSNR ~11.0, SSIM 0.41** — vs off-device baseline 25.9.
  Rendered eval view is an unrecognizable foggy blob (GT is a detailed desk).
- Root cause is the M0/M1 caps, not a pipeline bug: **500 iters (vs 30k) + random
  init (no seed) + only 16k splats**. Off-device needed 30k iters AND COLMAP
  seeding for 25.9; random-init off-device converges to only ~19 dB.
- Implication: need BOTH more iterations (30k @ ~5.5 it/s on Mali ≈ 90 min — a
  time/thermal problem, M5) AND seeding (M3, +6.75 dB, also converges faster).
  → confirms **M3 (on-device seeding)** as the right next lever.

## Seeding validation on-device (M3 justification) — DECISIVE
Restarted the GPU box (carbonite-noble distrobox was down → autostart service now
set up), pulled its existing `scan99_colmap` (COLMAP sparse: 21,599 points),
pushed to the Pixel. Brush reads COLMAP directly and auto-seeds from `points3D`.
Controlled test — same COLMAP poses, same 1000 iters, only init differs:

| Condition (COLMAP poses, 1000 iters, 720px) | PSNR | SSIM | time |
|---|---|---|---|
| **Seeded** (COLMAP points) | **21.51** | **0.82** | 177 s |
| **Random** init (points3D hidden) | 10.75 | 0.40 | 144 s |
| (ref) M1 ARCore random, 500 iters | ~11.0 | 0.41 | — |

**Seeding delta on-device = +10.8 dB / +0.42 SSIM** — bigger than off-device's
+6.75 dB (which was at 30k iters; at a phone-reasonable ~1000-iter budget random
init is nowhere near converged). **Conclusion: on-device seeding is essential,
not optional** — random init can't reach usable quality in phone time; seeding
hits PSNR 21.5 in ~3 min on Mali. → M3 is the priority next milestone.

Caveat: this used COLMAP poses+seed. Production seed must come from ARCore
(feature/Depth points, already in the ARCore pose frame) — that's the M3 build.
