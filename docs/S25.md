# Samsung S25 / Adreno support

Goal: run the full on-device pipeline on the Galaxy **S25** (Snapdragon 8 Elite,
**Adreno** GPU). Everything so far is verified on the Pixel 6 (**Mali**), so the
open risk is the **Adreno + wgpu/Vulkan compute** path (cubecl/burn training +
render kernels), which can differ by GPU vendor.

## What's already covered
- **ABI**: the app ships `arm64-v8a` (the S25's ABI) — no new build target needed.
- The 45 MB `libbrush_ffi.so` is arch-generic wgpu → any Vulkan-capable GPU.

## Adreno validation status (2026-07-07)
No S25 on hand. Tested on a **Samsung Galaxy Tab S8 (Adreno 730, Android 16)** as
a proxy.

**Verified on Adreno ✅**
- APK installs + runs; ARCore tracking, camera, and the **Depth API** all work.
- **Real-time GL rendering on Adreno** — the Polycam depth **coverage overlay**
  renders correctly (dense teal points, ~19k), with the capture pipeline running
  (46–59 keyframes, ~1700–2255 seed points, sharpness gate active).

**Blocked ⚠️ — could NOT validate the wgpu/cubecl training compute**
- The Tab S8 has a **pathological external-storage state**: writing a keyframe to
  the app's *own* `getExternalFilesDir()` fails with `EACCES (Permission denied)`
  (nested `captures/<ts>/images/*.jpg`), yet a write probe at the root can pass —
  so writes fail inconsistently and captures never persist ("No scans yet").
  Pushed datasets are also not readable by the app there.
- This is **device-specific** (Samsung + Android 16 + SD card, likely a
  Knox/enterprise or preview-OS storage quirk), **not** an Adreno/GPU issue and
  **not representative of the S25** (a phone with standard emulated storage).
- Net: training/render compute on Adreno remains **unproven** — but nothing
  observed suggests it won't work (GL rendering already runs on this Adreno).

## Robustness fix landed
- `Storage.kt` (`rumahkuRoot`): probe a **nested** write to external storage and
  fall back to **internal** (`filesDir`) if it fails. Fixes the clean
  "external genuinely unwritable → use internal" case for readers + writers
  (`CaptureSession`, `loadScans`). Note: it does **not** fully rescue the Tab S8,
  whose external probe passes but real writes still fail intermittently.

## Recommendation / next steps
1. **Validate on the actual S25** — a standard phone, no SD-card/Knox quirk. Install
   v0.1.0, reconstruct a dataset, confirm the compute path. This is the real target.
2. If Adreno training reveals kernel issues (cubecl/wgpu), likely fixes: shader
   workgroup limits, subgroup ops, or `wgpu` backend features — handle per-vendor.
3. Consider a "storage unavailable" user-facing error instead of silent
   `EACCES` capture failures (defensive, helps any odd device).
