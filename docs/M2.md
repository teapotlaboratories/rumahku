# M2 — Live / incremental training

Goal: **watch the splat form as you scan** (the "semi-real-time" promise). Today
the flow is capture → *then* train. M2 feeds keyframes into the trainer **during**
capture and shows the splat refining live.

## What Brush already gives us
Brush's training loop is built to stream: the dataset/scene can grow while
training runs, and the trainer emits the splats periodically (a `SplatsUpdated`
/ export signal). So the hard reconstruction math is done — M2 is about wiring
the on-device capture pipeline to a **long-lived streaming trainer** + a live
preview, instead of a one-shot `nativeTrain`.

## Architecture

```
CaptureSession (GL thread)          ReconstructionService-like streaming trainer
  keyframe (image, pose, intr) ─────►  push into trainer's dataset
  ARCore seed (colored) ────────────►  seed init (once, at start)
                                       trainer runs continuously (background)
  live preview  ◄─── nativeRenderView(current splat, a recent pose) ───┐
                                       current splats snapshot ─────────┘
```

### FFI (brush-ffi) — new streaming API
- `nativeStreamStart(seedPlyPath, outDir, maxRes)` → spins up a persistent trainer
  on the tokio runtime (reuse the device/runtime we already cache), seeded from
  the colored seed. Returns a handle/uses a global (like `CANCEL`).
- `nativeStreamPush(imageBytes/nv21, w, h, poseMatrix, flX, flY)` → add a training
  view; the trainer picks it up on its next step.
- `nativeStreamRenderPreview(poseMatrix, …) → ARGB` → render the **current** (live)
  splats from a pose (reuse `render_splats`, no reload — the splats live in the
  trainer). Cheap enough for a few fps.
- `nativeStreamFinish()` → stop, export final `.ply` to outDir (same artifact as
  `nativeTrain`), tear down.

### App
- A `StreamingReconstructionService` (or extend the existing service) owns the
  trainer for the capture's lifetime.
- `CaptureActivity`: while capturing, every Nth keyframe also calls
  `nativeStreamPush`; a small **live preview** (inset or toggle) shows
  `nativeStreamRenderPreview` from the current camera pose so you see it converge.
- On finish: `nativeStreamFinish` writes the `.ply`; the scan lands "Ready" —
  no separate reconstruct step.

## Risks / mitigations
- **GPU contention** — ARCore (camera + depth) + our GL overlays + wgpu training
  + preview render all hit the Mali/Adreno at once. Mitigate: throttle training
  steps, cap preview to ~2–4 fps, lower `maxRes` in live mode.
- **Thermals / battery** — continuous GPU. Reuse the M5 findings (2000 iters ≈
  7 min at 24–28 °C); live mode spreads compute over the scan, may run hotter.
  Watch temps; consider pausing training when tracking is lost.
- **Memory** — keyframes + growing splat + camera. Keep the streamed image res
  modest; cap frame count as today (`MAX_FRAMES`).
- **`burn_init_setup` once-per-process** — the streaming trainer must reuse the
  same device-init guard as `nativeTrain`/render (else SIGABRT).

## Build order (spikes)
1. **FFI spike**: `nativeStreamStart/Push/RenderPreview/Finish` over Brush's
   streaming trainer; prove it trains from pushed frames + renders the live splat.
   Verify off-device first (feed a known dataset frame-by-frame) then on Pixel 6.
2. **App**: streaming service + push keyframes during capture; final `.ply` on finish.
3. **UI**: live preview overlay while scanning; contention/thermal throttling.
4. Fallback: if live proves too hot/slow, keep it a **post-capture "refine live"**
   view (train after capture but stream the preview) — still shows convergence.
